# guide-readme

> README 작성 규칙. README 작성·수정 요청 시 로드.

## 작성 방식 — 스토리텔링

README는 항목 나열이 아니라 이야기로 쓴다.

문제를 발견했고 → 고민했고 → 이렇게 해결했다는 흐름이 자연스럽게 이어지도록 작성한다.
STAR(S/T/A/R) 구조를 섹션 레이블로 드러내는 것은 금지. 그 흐름을 산문으로 녹인다.
다이어그램·시각 자료는 설명의 흐름 중간에 자연스럽게 삽입. 별도 "Architecture" 챕터로 분리하지 않는다.
README가 끝날 때 독자가 "아, 이래서 만들었구나. 이렇게 동작하는구나"를 느껴야 한다.

**구조:** 도입(문제) → 전개(고민·설계) → 결말(결과)

## Mermaid 시각화

텍스트 설명만 쓰지 않고 Mermaid 다이어그램을 적극 활용한다.

| 용도 | 다이어그램 타입 |
|------|--------------|
| 아키텍처 설명 | `flowchart TD` / `graph TD` |
| API·데이터 흐름 | `sequenceDiagram` |
| 에이전트 역할 분리 | `flowchart LR` |
| 일정·단계 | `gantt` |
| ERD | `erDiagram` |
| 수치 비교 (TTL, 성능) | `xychart-beta` 바 차트 |
| 비율·구성 | `pie` |

**원칙:**
- 실제 데이터나 수치가 있는 섹션에만 추가 (억지로 넣지 않음)
- 구조 설명이 들어갈 때는 Mermaid 블록 우선 작성 후 텍스트 보완

## 버전 히스토리 자동 업데이트

git push(배포) 전후, `## 버전 히스토리` 테이블에 새 항목을 자동 추가한다. PO가 따로 요청하지 않아도 실행.

**형식:** `| vX.Y | YYYY-MM-DD | 한 줄 요약 |`

- 버전 번호: 기존 최신 버전에서 마이너(Y) +1
- 섹션이 없으면 먼저 만들고 추가
- 대상: 모든 프로젝트 (whalyx, cafe-inventory, resume-bhpark 등)
