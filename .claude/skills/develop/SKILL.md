# OMS 개발 워크플로우

## 트리거

PO가 새 기능 요청 또는 버그 수정을 요청할 때

## 실행 순서

### 1. PM — 요구사항 파악
- PO 요청을 태스크로 변환
- 영향 범위 파악 (어떤 에이전트/파일이 관련되는가)
- TPM에게 위임

### 2. TPM — 기술 계획
- 수정/생성할 파일 목록 결정
- 실행 순서 정의
- 의존성 충돌 여부 확인

### 3. 개발 실행
- 관련 파일 먼저 Read로 읽기 (절대 읽지 않고 수정하지 않는다)
- 코드 작성 / 수정
- 로컬 테스트 (curl 또는 python3 직접 실행)

### 4. 문서화
- 아키텍처 또는 파이프라인 변경 시 README 업데이트 필수

### 5. 커밋 & 푸시
```bash
git add [관련 파일]
git commit -m "feat/fix/docs: [내용]"
git push origin main
```

### 6. PM — PO 보고
- 완료된 작업 요약
- 잔여 이슈 명시
- 다음 스프린트 제안

## 절대 하지 않는 것

- PO에게 기술적 판단을 묻지 않는다
- 파일을 읽지 않고 수정하지 않는다
- 테스트 없이 완료 선언하지 않는다
- README 업데이트 없이 아키텍처 변경하지 않는다
