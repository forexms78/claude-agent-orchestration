# README 작성 가이드

## 핵심 원칙: 이야기로 써라

항목 나열 금지. **문제 발견 → 고민 → 해결**의 흐름이 자연스럽게 이어지도록 산문으로 작성.
독자가 읽고 나서 "아, 이래서 만들었구나. 이렇게 동작하는구나"를 자연스럽게 느껴야 한다.

**금지:**
- `S:`, `T:`, `A:`, `R:` STAR 구조를 섹션 레이블로 드러내는 것
- 별도 "Architecture" 챕터를 만들어 다이어그램만 모아두는 것
- 기능 목록만 나열하는 이력서 스타일

## 글 구조 (순서 권장)

```
1. 한 줄 소개 — 프로젝트가 뭘 하는지 한 문장
2. 왜 만들었나 (문제) — 불편함·공백·계기를 산문으로
3. 어떻게 해결했나 (설계·구현) — 고민과 선택 흐름 속에 Mermaid 삽입
4. 어떻게 동작하나 (결과·데모) — 스크린샷, 링크, 수치
5. 빠른 시작 — 로컬 실행 명령어
6. 버전 히스토리 — 배포 시 자동 업데이트
```

## Mermaid 다이어그램 — 흐름 중간에 자연 삽입

설명 흐름 안에 배치. 별도 챕터 금지.

| 상황 | 다이어그램 종류 |
|------|--------------|
| 전체 아키텍처 | `flowchart TD` 또는 `graph TD` |
| API·데이터 흐름 | `sequenceDiagram` |
| 에이전트 역할 분리 | `flowchart LR` |
| 단계·일정 | `gantt` |
| DB 구조 | `erDiagram` |
| 수치 비교 (성능, TTL 등) | `xychart-beta` 바 차트 |
| 비율·구성 | `pie` |
| 포지셔닝 | `quadrantChart` |

**원칙:** 실제 데이터·수치가 있는 섹션에만 삽입. 억지로 넣지 않는다.

## 버전 히스토리 — 배포 시 자동 업데이트

git push 전후 README의 `## 버전 히스토리` 테이블에 한 줄 추가. PO 별도 요청 없이 자동 실행.

```markdown
## 버전 히스토리

| 버전 | 날짜 | 변경 내용 |
|------|------|----------|
| v1.3 | 2026-04-07 | 새 기능 한 줄 요약 |
| v1.2 | 2026-03-26 | ... |
```

- 버전 번호: 기존 최신 마이너(Y) +1
- 섹션 없으면 먼저 만들고 추가
- 대상: 모든 프로젝트

## 문체
- "~했습니다" 대신 "~했다" (해체 또는 명사형)
- 짧은 문장. 한 문장 한 생각
- 코드 블록·표는 설명 흐름 중간에 자연스럽게
